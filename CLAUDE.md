# Hive — Project Notes for Claude

## AI Classification Rules

**Geography is part of the match, not just subject matter.**
"Food in Jayanagar" and "Food in HSR Layout" are different topics. The classification prompt must explicitly state that a topic only matches if both subject AND neighborhood align. Failing to enforce this causes clips to be incorrectly merged into geographically wrong topics.

**`community_new` must always result in a topic being created.**
If a clip is classified as `community_new` and no DB insert happens, the voice note ends up with `topic_id = null` and is invisible to the community feed. The edge function is responsible for creating the topic immediately at classification time.

**New topics require the user's location.**
The `transcribe-audio` edge function needs `lat`/`lng` in the request body to insert a new topic into the PostGIS `topics` table. `RecorderViewModel` must inject `LocationRepository` and call `getLastLocation()` before invoking transcription.

**Topic titles must be broad and categorical, not incident-specific.**
Bad: "New cafe on 12th main", "Pothole near Forum Mall"
Good: "Koramangala Food & Restaurants", "Road Conditions & Potholes"
The title generation prompt must explicitly instruct the LLM to produce a category-level title that 20+ people in the same area could contribute to. Niche titles fragment the community feed and kill the resonance effect.

## Navigation + Hilt ViewModel

**DO NOT rely solely on `savedStateHandle["argKey"]` in destination ViewModels.** Hilt can reuse a ViewModel across navigations to the same route pattern, causing the first topicId to always be used ("always Bellandur" bug).
**Fix:** Expose `loadForTopic(id: String)` and call it from `LaunchedEffect(topicId)` in the composable. Guard with `if (id == loadedTopicId) return`.

## AI Topic Summary

- Generated by `generate-topic-summary` edge function. Input: `{ topic_id }`. Calls Gemini (narrative + attribution) → Google Cloud TTS → stores MP3 at `summaries/{topicId}.mp3` in `audio-notes` bucket → upserts `topic_summaries` row.
- `GOOGLE_TTS_API_KEY` must be set in Supabase secrets. Falls back to Gemini key (TTS fails gracefully if wrong key).
- Segment timing is estimated at 150 WPM (not actual TTS timestamps). Android syncs by checking `currentPositionMs >= segment.startMs` via `AudioPlayer.currentPositionMs` (polled every 100ms).
- `topic_summaries` schema: `topic_id UNIQUE`, `audio_path`, `duration_seconds`, `segments JSONB [{text, start_ms, attributed_to}]`.

## Architecture Decisions

- Classification + topic creation happen entirely in the `transcribe-audio` Supabase Edge Function (not on the Android client). The Android app only receives the final result.
- `voice_count` on topics is computed via COUNT JOIN in the `get_popular_topics` and `get_topic_by_id` RPCs — there is no stored counter column. It reflects reality automatically as voices are linked.
- `LocationRepository.getLastLocation()` is a one-shot suspend fun (not a Flow). Use it when you need a single location snapshot (e.g., before uploading a voice note). Use `getLocationUpdates()` only for continuous tracking (map screen).
